# 設計書: 国立シティ 写実モノクロ・レンダラ（CityScene）

作成日: 2026-06-24
ステータス: ドラフト（ユーザーレビュー待ち）

## Context（なぜ作り直すのか）

国立駅周辺のVJビジュアルは、これまで **Canvas-2D のフラット陰影ボクセル**で実装してきた。しかし参照画像と並べると「いつまでも収束しない」根本原因が判明した：**画風（レンダリング技法）そのものが違う**。

- 参照画像のタッチ = **実在の建物データを、柔らかい光・AO（接触影）・滑らかなAA・微細な区画線・微かなグレインで“写実的に3Dレンダリングした”質感**。さらに**繊細な地形の起伏**が立体感を生んでいる。
- 現状 = フラット陰影のcanvas箱＋16段の硬いグレー＋ジャギー → 「図解（ダイアグラム）」。

ユーザーの結論：「Ikeda風／MinimalTechno風という概念」と「これまでの成果物」に執着せず、**参照画像のタッチに寄せる**。よって canvas-2D 路線を畳み、**WebGL（Three.js）＋実データ＋オフラインAOベイク**の新レンダラへ移行する。

本セッションで合意した方針（確定）:
1. パイプライン軸 = **生成リアルタイム＋iPad維持**（WebGLでベイク資産を軽く描画）。
2. データ源 = **実OSMデータ**（国立駅周辺の実建物・道路・区画）。
3. レンダ手法 = **AO/ライトをオフラインでベイク＋ポスト処理**（リアルタイム再ライティングはしない）。
4. **地形の起伏**（実DEM）を一次要素として導入。立体感の核。
5. リビール順 = **地形（格子）→ 道路 → 建物 → 木々**。
6. **旧国立駅舎**（1926年の三角屋根の保存建築＝市のモニュメント、現役JR駅とは別物）を実データに馴染ませつつ**最も目立つランドマーク**に。現役駅は控えめ・縮小可。
7. **主要道路**（大学通り・富士見通り・旭通り・学園通り＋中央線）は**埋没させない最上位の構造ライン**。富士見＝長辺、旭＝短辺の非対称を地理に忠実に。密度も実データ準拠で左右偏りなく。
8. 既存の **音声エンジン・スキャナHUD・PWA・操作UI** は流用。iPad ターゲット維持。モノクロ＋グレインのタッチ。

## 目標 / 非目標

**目標**: 参照画像の「タッチ」（写実モノクロ・地形起伏・柔らかいAO・微細な区画線・グレイン）に視覚的に到達した、国立駅周辺の生成的・音反応・WebGL都市スキャナを iPad PWA として動かす。

**非目標**: 完全フォトリアルな建築ビジュアライゼーション（iPad制約内で“一歩手前”まで）。他のVJシーン（Dancers/Tunnel等）の変更。配信中の2D地図の維持義務（新方針では役目を終える）。

## アーキテクチャ（3層）

### L1. オフライン素材ベイカー（ビルド時ツール `tools/citybake/`）
- Node + three.js の自己完結スクリプト（Blender不要・再現可能。高品質AOが要れば Blender 焼きに差し替え可能）。
- 入力: Overpass API（OSM）＋ GSI 基盤地図情報DEM（標高）。
- 出力: 最適化された**ベイク済みアセット**（glTF/バイナリ＋manifest）。**ここが「タッチ」の発生源**。

### L2. ランタイムWebGLレンダラ（`src/scenes/city/CityScene.js`, Three.js）
- ベイク資産をロードし、**安価な材質（ベイク頂点AO×光、リアルタイムライト無し）**で描画。
- ポスト処理（FXAA/MSAA＋フィルムグレイン＋トーン＋微ヘイズ＋ビネット）。
- 生成リビール＋音反応を駆動（visibility/scale/opacity/highlight、再ライティングしない）。

### L3. 既存シェル流用
- `AudioEngine`／`Clock`／スキャナHUD（`Overlay`＋シーン固有 drawHud）／`ControlPanel`／PWA。
- **WebGLキャンバスの上に2DのHUDキャンバスを重ねる**＝参照画像どおりの構成。`SceneManager` が 2Dシーン群と CityScene を切替。

## データモデル（ベイク済みアセット）

- **terrain**: DEM由来のヘイトフィールド・メッシュ（格子）。頂点に標高＋ベイクAO。垂直誇張は manifest パラメータで可変。
- **buildings**: マージ/インスタンス化した BufferGeometry。頂点カラー = AO×光。per-building 属性: id, 重心(u,v), 高さ, 駅からの距離キー(リビール順), 種別(generic / 旧駅舎 / 現役駅 / campus)。
- **roads**: 主要道路（大学・富士見・旭・学園・中央線）＝primary lines。区画/細街路＝secondary lines。`LineSegments`。
- **parcels**: 微細な区画線（地割り）。距離で減衰。
- **green/open**: 近黒の静かなゾーン（公園・緑地・一橋オープン）。
- **trees**: 並木（大学・富士見・旭）＋緑地。インスタンス（低ポリ/ソフト）。
- **landmark**: 旧国立駅舎（三角屋根が読める形、同一マテリアル、微ハイライト）。
- **manifest**: bbox, 投影原点, 垂直誇張, 駅座標, 主要道路ポリライン参照, 属性表記。

## L1 素材パイプラインの工程
1. **取得**: Overpass で国立bbox の building / highway / railway / leisure=park・landuse=grass / natural=tree(row)。GSI DEM タイル取得。
2. **前処理**: 建物押し出し（高さ = `building:levels`、無ければ用途・規模＋ノイズで低層推定）。道路を primary/secondary に分類。旧駅舎・現役駅・一橋・緑地を識別。旧駅舎は三角屋根が読めるよう微整形。
3. **地形 conform**: 建物・道路・木々を DEM 面に沿わせる。
4. **ベイク**: 半球レイキャストで **AO（地形の谷間・建物の接触影を含む）＋柔らかい指向光**を頂点カラーに焼く。
5. **出力**: 最適化 glTF/バイナリ＋manifest。**OSMは ODbL／GSIは出典明記が必要 → HUD隅に「© OpenStreetMap contributors / 地理院タイル」を小さく明記**。

## L2 ランタイム描画（タッチの再現）
- **材質**: ベイク頂点AO×光をそのまま出すモノクロ・シェーダ（ライト無し）。
- **AA**: FXAA（モバイル優先）／可能なら MSAA。
- **ライン**: 主要道路=明・連続、区画/細街路=従・低α、距離減衰。
- **ポスト（1パス）**: グレイン＋トーン/コントラスト＋微ヘイズ＋ビネット。
- **カメラ**: 参照に合わせた浅い俯瞰の oblique。中央線がほぼ水平・上部1/3、大学通りが下方向、フレーム幅いっぱい。

## アニメーション / 音反応
- **リビール順**: ① 地形が格子状に立ち上がる → ② 道路が走る（通電スイープ、主要道路は最上位で明） → ③ 建物が距離・音で rise → ④ 木々が点る → ⑤ HUD/微細。
- **音マップ**（既存思想流用、ベイク前提なので“再ライティングしない”）: Pad/低域=地形のブレス、主軸=道路通電/駅パルス(Kick)、Snare=交差点/エッジ、Hi-hat=木々/グレイン、Bass=密度の呼吸（不透明度/微発光）。
- **ドロップ/section変化**: 周辺レイヤの再リビール。旧駅舎はアンカーとして安定。

## iPad 性能戦略
- 建物はマージ/インスタンスで**数ドローコール**。AOは焼き込み（実行時AO無し）。材質はライト無し。ポストは1パス。
- 距離LOD（遠方は簡略）。`clock.quality` 連動でポスト簡略・本数間引き。60fps狙い、重い端末は自動退避。

## 既存アプリへの統合
- `CityScene` を新シーンとして追加。WebGLキャンバス＋2D HUDキャンバスの2層。`SceneManager` が切替。
- `AudioEngine`/`Clock`/`Overlay`/HUD/`ControlPanel`/PWA を流用。HUDのスキャナUI（参照と同一）を CityScene 用に再配置。
- 他のVJシーンと配信2D地図は**温存**。旧 canvas `GroundPlan` は置換 or 併存（実装時に判断）。
- 旧ポリシー「Canvas2DのみWebGL禁止」を**WebGL解禁**に更新（HANDOFF §6 を改訂）。

## スコープ / 本セッションの既存作業の扱い
- 今セッションで行った canvas `GroundPlan.js` の密度チューニング（STEP 0-5）は**新方針で役目を終える**。別ブランチ退避 or 破棄（ユーザー指示）。
- Three.js は新規依存（CDN/ローカル）として導入。

## リスクと対策
- **iPad WebGL 性能**: ベイク＋マージ＋ライト無し＋単一ポストで担保。退行時は LOD/quality 退避。
- **OSM データ完全性・高さ欠落**: 高さは推定（低層分布＋ノイズ）。参照も低層なので整合。
- **ベイク工程の複雑さ**: まず Node-three の最小ベイカーで通す。品質不足なら Blender 焼きへ。
- **タッチ忠実度**: AO＋地形起伏＋グレインが核。MVPで早期に視覚検証。
- **WebGL+Canvas 2層の整合**: HUDは2Dキャンバス、都市はWebGL、同一座標系のHUDアンカー投影を共有。

## 検証（どう確かめるか）
- **視覚**: 参照画像と並べ、タッチ（AO/起伏/グレイン/区画線）と構図（中央線水平・主要道路明・左右均等・旧駅舎強調）を確認。
- **リビール**: 地形→道路→建物→木々 の順を確認。
- **性能**: iPad（または同等）で 60fps、`quality` 退避が効くこと。
- **データ**: 主要道路（大学・富士見・旭・学園・中央線）が正確・連続・非埋没。属性表記あり。

## MVP の最小スコープ（最初の到達点）
1. L1 最小ベイカー: OSM建物＋主要道路＋簡易DEM起伏を取得→押し出し→頂点AOベイク→glTF出力。
2. L2 最小ランタイム: glTFロード＋モノクロ材質＋FXAA＋グレイン＋カメラ＝**1枚の静止“完成フレーム”**を参照に並べて検証。
3. リビール（地形→道路→建物→木々）＋音の最小接続。
4. HUD（既存スキャナUI）を重ねる。
5. その後、木々・緑地・旧駅舎作り込み・性能最適化・パラメータ調整。

## 未決 / 前提

- DEM 垂直誇張は可変パラメータ（実地形が平坦なので“繊細に読める”量に調整）。
